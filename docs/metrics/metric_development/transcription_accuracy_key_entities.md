# Transcription Accuracy: Key Entities

## Dataset Construction

Each conversation in EVA (when run on the cascade architecture) includes both the intended user speech generated by the user LLM and a transcription of that speech, representing what the assistant actually heard. For the initial dataset, we sample from these conversations and hand-label each entity along with whether it was correctly transcribed, capturing realistic transcription errors.

Importantly, we also include samples where transcription errors occur on non-entity text. This ensures the judge learns to distinguish errors that matter (those affecting key entities) from noise it should ignore.

Initial dataset stats: Across 38 conversations, 567 entities were identified, but only ~10% contained transcription errors.

## Data Augmentation

To address this imbalance, we augment the dataset with synthetically perturbed samples by applying targeted perturbations to correctly transcribed entities:

- Character drops in confirmation codes (e.g., ABC123 → ABC13)
- Phonetic swaps that mirror real ASR confusion patterns (e.g., 15 ↔ 50, A ↔ 8)
- Name variations that reflect plausible transcription drift (e.g., Steven → Stephen)

This brings the total error count to 181 out of 567 entities (~32%).

## Equivalence Tuning

We tune the judge prompt around two equivalence notions, while explicitly instructing it to ignore transcription errors on non-entity text:

- **Audio equivalence:** the entity sounds the same when spoken, even if the written text differs. Example: "mark dot lewis at gmail dot com" ≡ "mark.lewis@gmail.com"
- **Practical/semantic equivalence:** the entity was understood correctly, even if the formatting differs. Example: L E W I S ≡ L-E-W-I-S (user spells out their name letter by letter; transcription uses hyphens to separate the letters)

## Evaluation Metrics

To measure judge effectiveness, we compare the model generated JSON against human generated ground truth and measure the following key values: precision, recall, and correctness. Precision and recall are calculated on the identification of entities and correctness is calculated on the classification of entities.

- For identification measurement, we allow variations in whitespace and punctuation, as well as string boundaries only if the entire ground truth string is present in the judge's identified entity.
- Classification is simply whether or not the judge put true/false correctly, with a penalty applied if the entity could not be matched.

## Prompt Optimization

When optimizing our prompt using these metrics against our dataset, we saw very high recall and correctness (no penalty) with the initial prompt. By adding a section dedicated to what counts as an entity, we significantly raised precision (0.78 → 0.9) at very minor cost to recall (0.99 → 0.97).

## Manual Validation

Finally, judge outputs were manually reviewed on EVA runs for each sample across every simulation run to validate that the judge's decisions remain accurate under real evaluation conditions.
